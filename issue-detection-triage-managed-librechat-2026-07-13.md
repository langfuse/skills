## Verified non-issues

- **Answer quality when turns complete:** the sampled successful conversations show substantive, well-grounded analysis (e.g. a detailed month-over-month ClickHouse cost breakdown in [a53c2db1…](https://cloud.langfuse.com/project/cmo09hyuk01ttad079ljvue6w/traces/a53c2db19d320ba64733907861e5ff05)). No hallucination or refusal problems seen in the sample — but note this is unmeasured at scale (finding 1).
- **SQL execution reliability:** `run_select_query_mcp_ClickHouse` is by far the most-used tool (~20k calls in sample) with no execution-failure signatures surfaced; the ClickHouse MCP path itself looks healthy.
- **Custom user-defined agents:** faster (p90 99s) and cheaper (mean $0.28) than the default agent — no concentrated problem there.
- **Language matching:** in the sample, the agent replied in the user's language across PT/DE/FA/ZH/TR — no language-mismatch issue observed.

## Suggested order of action

1. **Finding 1 (monitoring)** — you are currently flying blind; stand up evaluators + user feedback first so every fix below is measurable.
2. **Findings 2 + 3 + 5 (latency / cost / duplicate calls)** — one root cause cluster (unbounded context + redundant loops); context compaction and tool memoization address all three and the cost is material ($2.6k/week).
3. **Finding 4 (hard failures)** — fix the pre-flight guards and the InvalidCharacter encoding bug; these leave real users with no answer.
4. **Findings 6, 7, 8 (loops, error serialization, observability)** — hygiene that makes the above durable and diagnosable.

## Open questions

- Is the `default` environment intentional, or should production be tagged (`environment=production`, `release=<sha>`) to separate it from any future staging traffic?
- Is tool-output redaction a hard compliance requirement? It currently blocks retrieval-relevance auditing (D14).
- Expected turn-latency SLO for this product? p90 of 233s is very high for interactive chat but may be acceptable for deep data-analysis sessions.

## Not covered

- **Retrieval / SQL-result relevance (D14):** tool outputs are redacted in traces, so I could not verify whether `search_docs`/`run_select_query` returned relevant content — only that calls succeeded. Requires finding 8(b) fixed, or an eval that runs inside the app.
- **Faithfulness / correctness of answers on customer data:** no ground truth and no scores; not evaluated at scale (this is finding 1).
- **PII/safety:** users paste real customer data (marketing spend, auth logs, org/brand IDs); no guardrail or PII scan is in place, but I did not do a dedicated safety sweep — worth a follow-up given the data sensitivity.
