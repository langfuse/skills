Verified non-issues

  • Answer quality when turns complete: sampled successful conversations show substantive, well-grounded analysis (e.g. a detailed month-over-month cost breakdown in a53c2db1). No hallucination/refusal seen in the sample — but unmeasured at scale (finding 1).
  • SQL execution reliability: run_select_query_mcp_ClickHouse (~20k calls in sample) shows no execution-failure signatures; the ClickHouse MCP path looks healthy.
  • Custom user-defined agents: faster (p90 99s) and cheaper (mean $0.28) than the default agent — no concentrated problem.
  • Language matching: the agent replied in the user's language across PT/DE/FA/ZH/TR — no mismatch observed.

Suggested order of action

  1. Finding 1 (monitoring) — you're flying blind; stand up evaluators + user feedback first so every fix below is measurable.
  2. Findings 2 + 3 + 5 (latency / cost / duplicate calls) — one root cluster (unbounded context + redundant loops); context compaction + tool memoization address all three, and the cost is material ($2.6k/week).
  3. Finding 4 (hard failures) — pre-flight guards + the InvalidCharacter encoding bug; these leave real users with no answer.
  4. Findings 6, 7, 8 (loops, error serialization, observability) — hygiene that makes the above durable and diagnosable.

Open questions

  • Is default environment intentional, or should production be tagged (environment=production, release=<sha>)?
  • Is tool-output redaction a hard compliance requirement? It currently blocks retrieval-relevance auditing.
  • Expected turn-latency SLO? p90 of 233s is very high for interactive chat but may be acceptable for deep data-analysis sessions.

Not covered

  • Retrieval / SQL-result relevance: tool outputs are redacted, so I could not verify returned content — only that calls succeeded. Needs finding 8(b) fixed, or an in-app eval.
  • Faithfulness / correctness at scale: no ground truth, no scores (this is finding 1).
  • PII/safety: users paste real customer data (marketing spend, auth logs, org/brand IDs); no guardrail or PII scan in place, and no dedicated safety sweep was run — worth a follow-up given the data sensitivity.
