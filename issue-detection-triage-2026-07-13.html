P0

1 · Prompt blows context limit

Reliability≥4 / 35 completions (11%)
Description
inline_code_completion fails with ValidationException: prompt is too long — 290k–342k input tokens per request. The completion feature is hard-broken for services with large schemas. Same pattern inflates query_construction/query_correction to ~645k total tokens (mostly cache reads, so cost is masked at ~$0.0015, but it sits at the context ceiling).
Root cause
The entire database schema is inlined into the prompt (<schema> block). Prompt size scales with customer schema size, unbounded, so any big-schema service exceeds the model’s context window.
Examples
ee8391be · 5d3701f8
Proposed fix
Code: cap/prune the schema block (relevant tables only via embedding or FK/usage heuristics, truncate column lists, hard token budget with graceful degradation instead of erroring).
Agent prompt (copy-paste)
The `inline_code_completion` prompt (and `query_construction`/`query_correction`) inlines the full ClickHouse schema in a <schema> block. On services with large schemas this reaches 342k input tokens and Bedrock rejects it with `ValidationException: prompt is too long` — the completion returns nothing (Langfuse traces ee8391beba6d7c981dd6c63ef103888f, 5d3701f85a98aca0e07ac9da4e8815e4, staging env). Add a token budget for the schema section: select only tables relevant to the current query context, truncate column lists past a limit, and fall back to schema-less completion rather than failing. Verify with a service whose schema serializes to >300k tokens.
